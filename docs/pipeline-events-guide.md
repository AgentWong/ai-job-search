# Pipeline Events Guide

How to manually add or edit rows in `job_search_log/pipeline_events.csv` to
track what happens after you apply. Designed for direct editing in
LibreOffice / a text editor — no scripts required.

## The two source-of-truth files

| File | What lives there |
|------|------------------|
| `job_search_log/applications.csv` | One row per application (metadata: company, role, date applied, source, match %, etc.) |
| `job_search_log/pipeline_events.csv` | Many rows per application — one row per pipeline event |
| `job_search_log/narratives.md` | Manual narrative notes per month (resume strategy changes, filter lifts, etc.) |

Generated artifacts (regenerate after every edit):

| File | Generated by |
|------|--------------|
| `job_search_log/dashboard.html` | `scripts/regenerate_dashboard.py` |
| `job_search_log/sankey_data.json` | `scripts/extract_sankey_data.py` |
| `job_search_log/job_search_sankey_d3.html` | `scripts/sankey_d3.py` |

## Schema: `pipeline_events.csv`

```
app_id, event_stage, event_date, event_outcome, event_notes
```

| Column | Description |
|--------|-------------|
| `app_id` | Composite key matching `applications.csv`. Format: `<company-slug>__<YYYY-MM-DD>`, e.g. `st-engineering-idirect__2026-04-04`. The slug is lowercase with non-alphanumeric chars replaced by `-`. |
| `event_stage` | Pipeline stage. Enum below. |
| `event_date` | `YYYY-MM-DD` of the event. May be blank if unknown, but should be filled when known. |
| `event_outcome` | Terminal outcome at this stage, or blank if you advanced past it. Enum below. |
| `event_notes` | Freeform notes (semicolon-separated for multiple bits). Surfaces as a tooltip in the dashboard. |

### `event_stage` enum

| Stage | When to use |
|-------|-------------|
| `applied` | The original application submission. Every app has exactly one of these. |
| `email_response` | Recruiter contacted you via email — outreach, scheduling a screen, AI/HackerRank/one-way assessment requests, "phone screen scheduled then cancelled". The screen has NOT actually happened yet. |
| `phone_screen` | A phone screen actually took place. **Do NOT use** for "scheduled" or "cancelled" — those are still `email_response`. |
| `interview_1` | First substantive interview round (video/in-person, technical or panel). |
| `interview_2` | Second round / "follow-up interview". |
| `interview_3`, `interview_4`, ... | Continue numerically for further rounds. The system scales to any N. |
| `offer` | A written offer was extended. |

Use sequential `interview_N` numbering regardless of how the company labels rounds. "Behavioral", "technical deep-dive", "panel" — they're all just N=1, N=2, N=3 in order.

### `event_outcome` enum

| Outcome | Meaning | Valid at stage |
|---------|---------|----------------|
| *(blank)* | You advanced to a later stage. A higher-stage row MUST exist for this app. | Any stage except the highest |
| `pending` | Currently at this stage; no terminal resolution yet. | Any non-`applied` stage |
| `no_response` | Applied with no contact at all. | `applied` only |
| `rejected` | Employer rejected at this stage. | Any |
| `ghosted` | Employer went silent at this stage with no follow-up. | Any |
| `withdrew` | You pulled out at this stage (e.g. declined an AI screener). | Any |
| `aborted` | You bailed more decisively (treat as `withdrew` if unsure). | Any |
| `accepted` / `declined` | Offer accepted / declined. | `offer` only |

## The cardinal rule

**The highest-numbered stage row determines where the application lives in the funnel.** Its outcome is the terminal state. All earlier stages should have blank outcome (= "advanced").

If an application's events look like this:

```csv
acme__2026-04-04,applied,2026-04-04,,
acme__2026-04-04,email_response,2026-04-15,,
acme__2026-04-04,phone_screen,2026-04-20,rejected,
```

— the funnel reports it as "rejected at phone_screen". The `applied` and `email_response` rows have blank outcomes because they were transit stages.

## Common scenarios — what to add

Every scenario starts from an existing application in `applications.csv` with one `applied` event in `pipeline_events.csv`:

```csv
acme__2026-04-04,applied,2026-04-04,no_response,
```

When something happens, you'll do TWO things:

1. **Append a new event row** for the new stage you reached.
2. **Update the prior highest row's outcome** — change it from a terminal state (e.g., `no_response`) to blank (since you've now advanced past it).

### Scenario 1: Recruiter emails you

The `applied` row was previously `no_response`; now it's `advanced`. Add an `email_response` row.

```csv
acme__2026-04-04,applied,2026-04-04,,                                      ← was 'no_response', now blank
acme__2026-04-04,email_response,2026-04-15,pending,Recruiter emailed
```

### Scenario 2: Recruiter ghosts after email

Same start as Scenario 1. Set the `email_response` outcome to `ghosted` once you give up on hearing back.

```csv
acme__2026-04-04,applied,2026-04-04,,
acme__2026-04-04,email_response,2026-04-15,ghosted,Phone screen scheduled then cancelled; no further contact
```

### Scenario 3: Phone screen happens

Add the `phone_screen` row, blank out the `email_response` outcome (since you advanced past it).

```csv
acme__2026-04-04,applied,2026-04-04,,
acme__2026-04-04,email_response,2026-04-15,,
acme__2026-04-04,phone_screen,2026-04-20,pending,
```

If you'd previously set `email_response` outcome to `pending`, swap it to blank.

### Scenario 4: Rejected after phone screen (no further interview)

```csv
acme__2026-04-04,applied,2026-04-04,,
acme__2026-04-04,email_response,2026-04-15,,
acme__2026-04-04,phone_screen,2026-04-20,rejected,Generic rejection email 2026-04-22
```

### Scenario 5: 1st interview happens

```csv
acme__2026-04-04,applied,2026-04-04,,
acme__2026-04-04,email_response,2026-04-15,,
acme__2026-04-04,phone_screen,2026-04-20,,
acme__2026-04-04,interview_1,2026-04-26,pending,
```

### Scenario 6: Interview round → next round

For each new round, add a row, blank out the prior outcome.

```csv
acme__2026-04-04,applied,2026-04-04,,
acme__2026-04-04,email_response,2026-04-15,,
acme__2026-04-04,phone_screen,2026-04-20,,
acme__2026-04-04,interview_1,2026-04-26,,
acme__2026-04-04,interview_2,2026-05-03,pending,
```

### Scenario 7: Offer extended

```csv
acme__2026-04-04,applied,2026-04-04,,
acme__2026-04-04,email_response,2026-04-15,,
acme__2026-04-04,phone_screen,2026-04-20,,
acme__2026-04-04,interview_1,2026-04-26,,
acme__2026-04-04,interview_2,2026-05-03,,
acme__2026-04-04,offer,2026-05-10,accepted,Signed 2026-05-12
```

Use `declined` instead of `accepted` if you turned it down.

### Scenario 8: You withdraw at email stage (e.g. AI screener requested)

```csv
acme__2026-04-04,applied,2026-04-04,,
acme__2026-04-04,email_response,2026-04-09,withdrew,Requested one-way AI screener; declined
```

Same pattern at any stage — set the highest stage's outcome to `withdrew`.

### Scenario 9: Skipped a stage in your records (e.g. went straight to interview_1)

If a phone screen happened but you didn't log it, just add the missing row when you remember:

```csv
acme__2026-04-04,applied,2026-04-04,,
acme__2026-04-04,email_response,2026-04-15,,
acme__2026-04-04,phone_screen,2026-04-20,,                                  ← backfilled later
acme__2026-04-04,interview_1,2026-04-26,pending,
```

The `event_date` for the missing row can be blank if you don't know it. The Sankey only cares about which stages were reached, not the dates.

### Scenario 10: Multiple apps to the same company (rare)

If you applied twice to the same company on the same date (separate roles, etc.), the second `app_id` is suffixed:

```
acme__2026-04-04
acme__2026-04-04-2
```

Each has its own set of events.

## Editing tips

- **Keep rows sorted by `app_id` then by stage order.** The dashboard and sankey scripts will re-sort, but humans grep this file too. Order: `applied` → `email_response` → `phone_screen` → `interview_1` → `interview_2` → … → `offer`.
- **Empty-string fields are fine.** A blank `event_date` or `event_notes` is valid CSV (just two consecutive commas).
- **Quoting:** LibreOffice will quote any field containing a comma, double-quote, or newline. The scripts use `csv.QUOTE_MINIMAL`, which matches LibreOffice's default behavior.
- **Don't delete an `applied` row.** Every app in `applications.csv` must have exactly one `applied` event in `pipeline_events.csv`. Delete the application instead if you want to remove it.
- **Don't add duplicate `(app_id, event_stage)` rows.** Each pair should appear at most once. To "update" a stage, edit the existing row in place.

## Finding an `app_id` quickly

The `app_id` is `<slugified-company>__<YYYY-MM-DD-applied>`. To slugify by hand:

1. Lowercase the company name.
2. Replace runs of non-alphanumeric chars with a single `-`.
3. Strip leading/trailing `-`.

Examples:
- `Acme Corp` → `acme-corp`
- `ST Engineering iDirect` → `st-engineering-idirect`
- `Aces Risk Management LLC` → `aces-risk-management-llc`
- `D&H Distributing` → `d-h-distributing`

If unsure, search `applications.csv` by company name to copy the existing `app_id`.

## After editing

Regenerate the dashboard and sankey:

```
.venv/bin/python scripts/regenerate_dashboard.py
.venv/bin/python scripts/extract_sankey_data.py
.venv/bin/python scripts/sankey_d3.py
```

Then open `job_search_log/dashboard.html` and `job_search_log/job_search_sankey_d3.html` to verify the edits flowed through.

## When to use the helper scripts vs. direct CSV edit

Both work. The helper scripts validate inputs and prevent dumb mistakes (duplicate stage rows, malformed dates, unknown outcomes). Direct CSV editing is faster when you're already in LibreOffice processing a batch of email updates.

| Action | Script | Direct edit |
|--------|--------|-------------|
| Register a new application | `scripts/append_application.py` | Add a row to `applications.csv` AND a row to `pipeline_events.csv` |
| Add a progression event | `scripts/append_event.py <APP_ID> <stage> --date YYYY-MM-DD [--outcome ...]` | Add a row to `pipeline_events.csv`, blank out the prior highest-stage outcome |
| Set/change a terminal outcome | `scripts/append_event.py <APP_ID> --update-outcome <outcome>` | Edit the highest-stage row's `event_outcome` cell |

The `process-clippings` workflow uses `append_application.py` automatically when adding new clippings; everything else is yours to do as you prefer.
